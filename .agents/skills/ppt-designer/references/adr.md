## ADR-001: 무드보드 수집을 필수 단계로 설정

**날짜:** 2026-04-02
**상태:** 채택

**맥락:** AI가 PPT를 만들 때 매번 일반적인 판단을 해서 "AI스러운" 결과물이 나옴. 유저마다 원하는 무드가 다른데 이걸 반영할 구조가 없었음.

**결정:** 슬라이드 생성 전에 반드시 무드보드(레퍼런스 이미지, 키워드, 컬러 방향)를 수집하고, 여기서 디자인 토큰을 추출한 뒤 유저 확인을 받는 플로우를 강제.

**근거:** Notion 회고 문서("AI로 디자인 시스템 구축하기")의 핵심 인사이트 — "AI에게 좋은 디자인을 만들어달라고 하지 말고, AI가 틀리기 어려운 구조를 먼저 만들어라." 토큰이 있으면 AI의 판단 품질이 달라짐.

## ADR-002: 타이틀 앞 세로 accent bar(`|`) 금지

**날짜:** 2026-04-02 (iteration-2)
**상태:** 채택

**맥락:** iteration-1 결과물에서 모든 콘텐츠 슬라이드에 타이틀 왼쪽에 세로 파란 바를 넣었음. 유저 피드백: "굳이 | 라인을 넣지 않아도 돼."

**결정:** 안티패턴 #1으로 지정. 타이틀 앞에 세로 accent bar를 넣지 않는다.

**근거:** 레퍼런스로 제공된 실제 발표자료에는 타이틀 앞에 세로 바가 없음. 이것은 AI가 "시각적 장식"으로 추가하는 패턴이지 인간 디자이너의 관행이 아님. 폰트 weight와 크기만으로 충분히 위계가 잡힘.

## ADR-003: 한국어 타이틀 크기 28-32pt → 22-26pt로 축소

**날짜:** 2026-04-02 (iteration-2)
**상태:** 채택

**맥락:** iteration-1에서 타이틀이 28-32pt으로 나왔는데, 유저 피드백: "텍스트가 쓸데없이 크고 AI스러워." 실제 발표자료 레퍼런스와 비교하면 확실히 큼.

**결정:** 한국어 슬라이드 타이틀 권장 크기를 22-26pt로 변경. KPI 숫자도 28-32pt → 24-28pt.

**근거:** 한글은 Latin보다 획이 많고 시각적으로 무거움. 같은 pt에서 더 크게 느껴짐. 실제 한국 발표자료 레퍼런스들을 보면 타이틀이 22-26pt 범위. 36pt+ 타이틀은 배너 광고처럼 보임.

## ADR-004: 카드형 디자인 줄이기, 플랫 레이아웃 우선

**날짜:** 2026-04-02 (iteration-2)
**상태:** 채택

**맥락:** iteration-1에서 거의 모든 콘텐츠를 카드(둥근 사각형 + 그림자)로 감쌌음. 유저 피드백: "실제로는 카드 형태의 디자인을 많이 사용하지는 않는 것 같아."

**결정:** 카드 사용을 최소화. 텍스트 위계(폰트 크기, weight, 색상)와 얇은 구분선으로 구조를 잡는 플랫 레이아웃을 기본으로.

**근거:** 실제 발표자료 레퍼런스를 보면 카드는 특정 강조 영역에만 사용되고, 대부분의 콘텐츠는 텍스트 + 구분선 + 여백으로 구성됨. 카드를 과도하게 사용하면 UI 컴포넌트처럼 보여서 "앱 화면" 느낌이 남.

## ADR-005: 슬라이드 마진 축소 (0.7" → 0.55")

**날짜:** 2026-04-02 (iteration-2)
**상태:** 채택

**맥락:** iteration-1에서 좌우 마진이 0.7"였는데, 유저 피드백: "페이지의 width를 조금 더 넓게 써도 될 것 같고."

**결정:** 좌우 마진을 0.55"로 축소. 콘텐츠 영역 8.6" → 8.9".

**근거:** 레퍼런스 슬라이드들은 대부분 화면을 넓게 활용함. 0.7"는 안전하지만 콘텐츠가 좁아 보이는 부작용이 있었음. 0.55"면 여전히 충분한 safe zone이면서 더 넓은 레이아웃 가능.

## ADR-006: Fluent Emoji 3D를 기본 아이콘 소스로 채택

**날짜:** 2026-04-02 (iteration-2)
**상태:** 채택

**맥락:** iteration-1에서 react-icons로 아이콘을 렌더링했는데, 결과가 단색 평면 아이콘이라 시각적 풍성함이 부족. 유저: "icon 같은 걸 사용은 못하나? animated-fluent-emoji, tossface 같은 거."

**결정:** Microsoft Fluent Emoji 3D PNG를 기본 아이콘으로 사용. GitHub CDN에서 빌드 시점에 다운로드.
- URL 패턴: `https://raw.githubusercontent.com/microsoft/fluentui-emoji/main/assets/{Name}/3D/{name_snake}_3d.png`
- react-icons는 미니멀/기업용 무드일 때 대안으로 유지.

**근거:** Fluent Emoji 3D는 고퀄리티 래스터 이미지로, PPT에 넣으면 시각적 완성도가 크게 올라감. MIT 라이센스. GitHub raw URL로 직접 다운로드 가능해서 별도 의존성 없음.

## ADR-009: 카드 사용 기준 — "내용 형태가 다를 때만"

**날짜:** 2026-04-02 (iteration-2 추가 학습)
**상태:** 채택

**맥락:** ADR-004에서 카드를 줄이라고 했지만, 레퍼런스에서 카드를 잘 쓴 사례도 있음. 한 레퍼런스(3컬럼 카드)에서 각 카드 안에 바차트, 강조박스, 인용카드가 들어가 있어서 카드가 자연스러움. 다른 레퍼런스에서 2컬럼 카드 + 이모지가 visual anchor로 효과적.

**결정:** 카드 사용을 무조건 금지하지 않되, 사용 기준을 명확히 함:
- 각 카드가 **서로 다른 유형의 콘텐츠**를 담을 때 사용 (차트 vs 인용 vs 통계)
- 동일한 텍스트 리스트 3개를 카드로 감싸는 건 금지
- 다크 배경에서는 border/stroke 카드, 라이트 배경에서는 subtle fill
- 카드 안에 이모지/아이콘을 visual anchor로 넣으면 효과적

**근거:** 카드 자체가 문제가 아니라, AI가 카드를 "모든 콘텐츠의 기본 컨테이너"로 쓰는 것이 문제. 인간 디자이너는 카드로 "이건 독립적인 단위"라는 시각적 신호를 줌. 내용이 실제로 독립적이고 형태가 다를 때만 카드가 의미 있음.

## ADR-010: 카드 내부 패딩 최소 0.25" → 0.3" 권장

**날짜:** 2026-04-02 (iteration-3)
**상태:** 채택

**맥락:** iteration-2에서 카드를 쓴 슬라이드가 나왔는데, 카드 내부 콘텐츠가 테두리에 너무 가까움. 유저: "겉의 padding이 조금더 있으면 좋겠네."

**결정:** 카드 내부 패딩을 최소 0.25", 권장 0.3"으로 상향. content.x = card.x + 0.3, content.w = card.w - 0.6.

**근거:** 카드가 있으면 콘텐츠가 테두리와 충분한 거리를 가져야 "답답하지 않은" 느낌이 남. 0.2"는 부족함이 느껴짐.

## ADR-011: PptxGenJS bullet indent 기본값 오버라이드

**날짜:** 2026-04-02 (iteration-3)
**상태:** 채택

**맥락:** iteration-2에서 불릿 리스트의 점(·)과 텍스트 사이 간격이 너무 넓음. 유저: "리스트의 점과 텍스트 간 간격이 너무 멀어."

**결정:** 모든 불릿 리스트에서 `{ bullet: { indent: 10 }, indentLevel: 0 }` 사용. 기본 indent(~20pt)를 쓰지 않음.

**근거:** PptxGenJS 기본 bullet indent가 ~20pt로, 한국어 텍스트에서 점과 텍스트 사이가 시각적으로 분리되어 보임. 10pt가 적절한 간격.

## ADR-012: Fluent Emoji 다운로드 시 유효성 검증 필수

**날짜:** 2026-04-02 (iteration-3)
**상태:** 채택

**맥락:** iteration-2에서 일부 아이콘이 깨진 채로 슬라이드에 들어감. 유저: "가끔 깨지는 이미지가 있어."

**결정:** 이미지 다운로드 후 반드시 유효성 검증:
1. 파일 존재 확인
2. 파일 크기 > 1KB
3. PNG 헤더 확인 (89 50 4E 47)
4. 실패 시 텍스트 이모지로 대체 또는 스킵

**근거:** Fluent Emoji GitHub repo의 파일명 패턴이 일관적이지 않아서 (공백, 대소문자, 특수문자) URL을 정확히 맞추기 어려운 경우가 있음. 깨진 이미지는 전체 슬라이드 품질을 크게 떨어뜨림.

## ADR-013: 섹션 간 최소 간격 규칙 추가

**날짜:** 2026-04-02 (iteration-3)
**상태:** 채택

**맥락:** iteration-2에서 description↔table, title↔table 간격이 너무 가까운 슬라이드가 있었음. 유저: "description과 테이블 간의 간격이 너무 가까워."

**결정:** 섹션 간 최소 간격 규칙:
- 타이틀 블록 → 첫 콘텐츠: 0.3"

**근거:** 텍스트와 데이터 시각화 사이에 숨 쉴 공간이 없으면 정보가 뭉쳐 보임. 특히 한국어 텍스트는 시각적 밀도가 높아서 간격이 더 중요.

## ADR-014: 플로우 다이어그램은 박스로 감쌀 것

**날짜:** 2026-04-02 (iteration-3)
**상태:** 채택

**맥락:** iteration-2에서 "Step 1 → Step 2 → Step 3" 같은 플로우를 텍스트+화살표로만 표현. 유저: "오히려 박스 형태로 감쌌으면 더 좋았을 것 같아."

**결정:** 플로우/프로세스를 표현할 때 각 단계를 ROUNDED_RECTANGLE로 감싸고 화살표로 연결. 텍스트만 나열하지 않음.

**근거:** 플로우는 시각적으로 "단계"가 구분되어야 읽기 좋음. 텍스트+화살표만으로는 단계의 경계가 모호함.

## ADR-015: 이미지 배치 — 라이트는 라운드, 다크는 직사각형, border 금지

**날짜:** 2026-04-02 (iteration-3 피드백)
**상태:** 채택

**맥락:** iteration-3에서 이미지 사용이 "최악"이라는 피드백. 이미지가 슬라이드에 아무 맥락 없이 올라가거나, 배치 방식이 어색했음.

**결정:**
- 라이트 배경: rounded container (subtle fill, border 없음) 안에 이미지 배치. 또는 이미지를 직접 round로.
- 다크 배경: 직사각형으로 딱딱하게 배치. round 없음.
- border/stroke는 절대 사용하지 않음.
- 이미지 edge는 콘텐츠 마진에 맞춤.

**근거:** 레퍼런스 분석 결과:
- iPhone 슬라이드: 라이트 bg + rounded image container (no border, subtle fill)
- 레퍼런스 A: 라이트 bg + 이미지 직접 배치 (no round, clean)
- 레퍼런스 B: 라이트 bg + gray rounded container with screenshots
- 레퍼런스 C: 다크 bg + 이미지 직사각형 (no round = rigid feel)

border를 넣는 순간 "파워포인트 기본 스타일" 느낌이 남. 프로 디자이너는 이미지에 border를 넣지 않음.

## ADR-016: 데이터 슬라이드에 장식용 이미지 금지

**날짜:** 2026-04-02 (iteration-3 피드백)
**상태:** 채택

**맥락:** iteration-3 Eval 2 슬라이드 4(KPI 3x2 그리드)의 우상단에 green swirl 이미지가 배치됨. 유저: "이런식의 너무 뜬금없는 이미지는 최악이야."

**결정:** KPI 그리드, 테이블, 비교 차트 등 데이터 중심 슬라이드에는 장식용 이미지를 넣지 않는다. 이미지는 커버, Layout B(텍스트+비주얼), 섹션 디바이더에만 사용.

**근거:** 데이터 슬라이드는 숫자와 라벨이 주인공. 이미지를 넣으면 공간을 뺏기고, 시선이 분산되고, "이미지가 남아서 억지로 넣었다"는 느낌을 줌. 이미지가 내용을 직접 설명하지 않으면 넣지 않는 것이 프로.
